# Notes App (app:notes)

Last verified: 2026-06-02 (boox-ink-backend)

## Purpose
Main application module that wires together ink rendering, storage, and user interaction into a single-activity note-taking experience optimized for e-ink. Includes a nav bar for moving between pages, plus a full-screen Library overlay for browsing folders and notebooks as a card grid.

## Contracts
- **Exposes**: `MainActivity` (entry point), `DrawView` (drawing surface), `ToolBar` (tool selection), `PageNavigationLogic` (pure page index/bounds/label math), `ToolSelectionLogic` (pure tool/variant state), `LassoSelectionLogic` (pure selection geometry — strokes AND text boxes), `Clipboard`/`InProcessClipboard` (mixed-content `ClipboardPayload`), `TextBoxSerializer` (defensive JSON codec; shelf-ready for B1 `app_state.clipboard_json` round-trip — no callers yet), `SelectionMenuView`, `SettingsView` (full-screen Settings overlay), `SettingsFormLogic` (pure pitch-radio rules), `LibraryView` (full-screen Library overlay; `Callbacks.onOpenSearch` + public `navigateToFolder(folderId?)`), `RecycleBinView` (full-screen Recycle Bin overlay), `LibraryAdapter`/`LibraryItem` (sealed `Folder`/`Notebook`), `BreadcrumbView`, `LaunchLogic`/`BreadcrumbLogic`/`NotebookNameParser`/`RelativeTime`/`ThumbnailCacheLogic` (pure helpers), `ThumbnailRenderer`/`ThumbnailCache`/`ThumbnailLoader`, `TextBoxEditOverlay` (full-screen edit modal — subsumes the legacy in-canvas `TextBoxEditor` + the standalone Options dialog), `TextBoxEditOverlayLogic` (pure commit/cancel decisions), `TextBoxMenuView` (Edit/Options/Delete pill — both Edit and Options open the overlay, differing only in initial focus), `FontCatalog` (/system/fonts enumeration + fallback), `search.SearchDialog` (modal AlertDialog over the Library) + `search.SearchResultsAdapter`, `ocr.OcrTextDialog` (editor modal showing server-recognized text with a forward-compatible "Device recognized" stub entry in the source dropdown), `caldav.*` (lasso-recognize → CalDAV VTODO flow with offline outbox/drainer — see Key Decisions: `VTodoBuilder`/`VTodoInput`/`VTodoDue`, `CalDavClient` + `OkHttpCalDavClient` + `CalDavCredentials` + `CalDavResult`, `SecureCredentialsStore` + `EncryptedPrefsCredentialsBackend` + `SyncCredentials`, `CalDavTaskSheetLogic` + `DueChoice`, `CalDavTaskSheet`, `CalDavOutboxStore` (drainer-facing interface; bridged to `NotebookStore` single-writer thread), `CalDavOutboxLogic` + sealed `DrainOutcome` (Success / RetryLater / DeadLetter — pure retry policy), `CalDavOutboxDrainer` + nested `DrainerStatus`/`Phase` + sealed `TryOutcome` (Sent / Queued), `NetworkAvailabilityMonitor`)
- **Guarantees**: Strokes auto-save on pen-up. Strokes restore on app launch (non-blocking async load), reopening the last-active notebook+page; if there is no resumable notebook (blank/empty active id or empty library, per `LaunchLogic`) the app opens into the Library instead. Backend lifecycle managed across pause/resume. Finger input rejected (stylus/eraser only). Page/notebook switching commits the current stroke (via clearAll) and refreshes the canvas to avoid e-ink ghosting. Lasso selects strokes by centroid; Cut/Copy/Delete via a floating pill + in-process clipboard; Paste is tap-to-place (arm, then tap the canvas); a closed selection can be dragged to move it in place. The Library lists folders-then-notebooks inside the current folder (null = root) as a 4-column card grid; tap a notebook to open it, tap a folder to descend, breadcrumb/back to walk up.
- **Expects**: `core:ink` for backend and stroke types, `core:format` for persistence.

## Dependencies
- **Uses**: `core:ink` (InkBackend, BackendDetector, PageTransform, Stroke, Tool, StrokeGeometry, PressureCurve), `core:format` (NotebookRepository + NotebookMeta/PageMeta/FolderMeta/NotebookCard/FolderCard/FolderPathLogic, owned exclusively by `NotebookStore`), `core:sync` (just the pure `SyncBackoff`/`SyncJoinPlan` policy now) + the **RhizomeSync library** `io.rhizome.core` (`SyncEngine`/`SyncConfig`/`SyncStatus`/`SyncResult`/`SyncTransport`) + `io.rhizome.http` (`HttpUrlTransport`) — all driven by `SyncController`, `androidx.recyclerview` (Library card grid only), kotlinx-coroutines (sync controller scope), `com.squareup.okhttp3:okhttp` (CalDAV PUTs only — `caldav.OkHttpCalDavClient`), `androidx.security:security-crypto` (EncryptedSharedPreferences backing `caldav.SecureCredentialsStore`)
- **Used by**: Nothing (top-level app module)
- **Boundary**: Should not contain domain logic; delegates to core modules

## Key Decisions
- Activity (not Fragment/Compose): minimal overhead for single-screen e-ink app. Secondary screens (Settings, Library) are full-screen overlay Views added to `android.R.id.content`, not new Activities — keeps the single `NotebookStore` (single-writer DB invariant) and avoids fragments. The Library is reached by tapping the fixed-width Library grid icon in the nav bar (a fixed icon, NOT the variable-width notebook name, so different name lengths never reflow the bar; the old `showNotebookPicker` dialog is REMOVED); its own folder/notebook/create dialogs are driven by `MainActivity` via `LibraryView.Callbacks`
- **Toolbar caption convention (both the editor toolbar and the Library header):** every action is a vertical icon-over-caption cell where the cell is the clickable hitbox and the inner `ImageButton` is non-clickable (taps fall through). A trailing **`▾`** on the caption means "opens a chooser/dialog" (editor: `Fountain ▾`/`Stroke ▾`/`Template ▾`; Library: `Folder ▾`/`Notebook ▾`); a plain caption is a direct action (`Lasso`/`Clear`/`Refresh`/`Settings`/`Up`). Captions are static strings (fixed widths → no reflow). The editor toolbar is 30dp; the Library header is 44dp (icon+caption + breadcrumb needs the extra room)
- RecyclerView is used only by the Library card grid (`androidx.recyclerview` — the app's first and only RecyclerView). Thumbnails render off-thread (`ThumbnailLoader`) into an LRU `ThumbnailCache`; eviction/sizing math is pure in `ThumbnailCacheLogic`
- DrawView owns offscreen bitmap: all strokes rendered to bitmap, blitted to screen in onDraw. A single `composeStaticBitmap()` defines page layering — template → BOTTOM-band text boxes → ink → TOP-band text boxes — and every rebuild path funnels through it (onSizeChanged/mergeLoadedStrokes/mergeLoadedTextBoxes/clearAll/redrawBitmap); `setTemplate(effectiveTemplate, pitchMm)` repaints on change
- **Text boxes (Tool.Text):** z-ordered text elements rendered with `StaticLayout`, clipped to the box (overflow retained in the model, not drawn). Drag-to-draw places a box; the full-screen `TextBoxEditOverlay` (attached to `android.R.id.content`, opaque white) edits it — text + font + size + weight + border + z-band all in one screen. The overlay obscures the canvas, so the soft keyboard's pan/resize no longer ghosts on the e-ink panel ([[viwoods-adjustnothing-anr]] proved manifest tweaks are a dead end on Viwoods). A freshly drag-drawn box is held as `pendingNewBox` (not yet in `textBoxes`, not persisted) — Cancel discards it cleanly with no DB write. `transformingBoxId` boxes are skipped in the static bitmap (shown live in `onDraw`). The overlay's `commitIfShowing()` fires from tool/page/notebook switch, library open, and onPause; back-button calls `requestCancel()`. On Done, `DrawView.commitOverlayBox` recomputes the rendered height via `measureTextBoxHeightPx` (same StaticLayout shape as `drawTextBox`) and persists. Empty text on Done routes to discard (pending-new gone, existing-box soft-deleted). Tap a box to select (corner handles + `TextBoxMenuView` pill); drag interior = move, drag handle = resize (reflows live). Both pill buttons (Edit, Options) open the same overlay — Edit focuses the EditText so the keyboard pops, Options leaves it defocused so the style strip is the first interaction. Font/size catalog is `FontCatalog` (off-thread `/system/fonts` scan; resolves a stored basename to a Typeface with a default fallback so a synced box whose font is absent still renders). Active style persists in `Settings.textFontName`/`textFontSizeV`. Text-box SYNC is gated off client-side (see core:format) — boxes persist + render locally now. Template pitch uses true panel PPI (293 on the Mini, set in MainActivity by isEInk), NOT densityDpi/xdpi
- Hardware eraser (TOOL_TYPE_ERASER) always erases; it honors the last-picked eraser variant (Stroke/Pixel, default Stroke) rather than hard-defaulting to Stroke — `DrawView.activeTool`'s setter records eraser picks into `lastEraserVariant`, and `handleEraser` uses it when the active tool isn't itself an eraser. Both variants use a 40f tip (they differ in behavior — whole-stroke wipe vs pixel rub — not size)
- **Boox/Onyx firmware-ink integration (branch `feat/boox-ink-backend`):** when `backend.ownsInput()` (Onyx), the firmware owns the stylus and renders live ink directly to the panel; `app:notes` wires it up and adapts the UI:
  - **Canvas-only `SurfaceView` topology** — `onCreate` adds a bare `SurfaceView` BEHIND `DrawView` inside `canvas_container` (which already sits below the navbar) and `backend.attachInput(surface, drawView.inputStrokeSink(), emptyList())`. DrawView goes `TRANSPARENT` so the reconciled page bitmap on the surface shows through; the navbar is a sibling OUTSIDE the firmware-owned region so it refreshes normally (no exclude rect, no global freeze on toolbar taps). `canvasTopOffset` collapses to 0 (surface co-extensive with DrawView).
  - **Tool routing** — the toolbar's `onToolSelected` calls `backend.setActiveTool(tool)` (firmware owns Pen only; every other tool hands the stylus back to ordinary MotionEvent dispatch) and, on an ACTUAL tool change, `drawView.refreshPanelForUi()` (the freeze-toggle reconcile that repaints the toolbar's new selected-state — live firmware render otherwise suppresses normal EPD posting). `handleStylus` swallows the duplicate stylus MotionEvent when `ownsInput() && activeTool is Pen` (firmware is the sole Pen path → no double-persist).
  - **Pen settings popup draw-to-dismiss** — `onPopupVisibilityChanged → setInputSuspended`, `onCoexistPopupBounds → setOverlayExcludeScreenRect` (carve the popup out of firmware capture so its buttons work while ink stays live elsewhere), `setOnFirmwarePenDown { dismissActivePopup() }` (starting to draw closes the menu).
  - **Overlay/dialog visibility discipline (the "frozen except drawing" fix, `60ee6ad`)** — live firmware render globally suppresses normal EPD UI posting, so an editor-obscuring surface opened while on Pen would be added to the view tree but never painted — INVISIBLE, on top, eating every touch (only system Back recovered). Two coordinated mechanisms: (1) **content overlays** — `openLibrary()` calls `backend.setInputSuspended(true)` BEFORE `libraryView.show()` (drop render before it draws) and `closeLibrary()` resumes; Settings/RecycleBin inherit the Library's suspend (opened from it — do NOT toggle them or you desync on return); Text/CalDav surfaces are already covered because their tools dropped render via `setActiveTool`. (2) **dialogs** — every `AlertDialog` steals window focus, so `onWindowFocusChanged` is the single choke point: drop render on focus-loss, restore on regain gated on `!anyEditorObscuringOverlayShowing()` (a dialog stacked over the Library returns to the still-suspended Library, not a re-wedge). On dialog dismiss, `backend.cleanNextReconcile()` makes the editor repaint a one-shot GC (the dialog's high-contrast text ghosts through the default low-flash mono reconcile). All no-ops on Viwoods/Generic (`ownsInput()==false`).
- 900ms postDelayed invalidate after pen-up: triggers quality e-ink refresh without excessive redraws
- All persistence flows through `NotebookStore` (single background thread); DrawView/MainActivity never touch `NotebookRepository` or the DB directly. Save is fire-and-forget; erase reconcile + load run off-thread and post results back to the main thread
- **CalDAV tasks + secret storage:** sync credentials and CalDAV credentials live in `EncryptedSharedPreferences` (`forestnote_secrets`, AES-256 in Android Keystore), owned by `caldav.SecureCredentialsStore` over a `KeyValueBackend` interface (production: `EncryptedPrefsCredentialsBackend`). `Settings.syncUsername`/`syncPassword` data-class fields are kept for backward compatibility (no SQLDelight schema change) but are **drained to empty strings** by a one-shot migration in `MainActivity.onCreate` (`SecureCredentialsStore.migrateSyncCredsFromSettings → MigrationResult.Migrated`); readers MUST go through ESP afterwards. `SyncController.resolveCredentials` reads ESP first, falls back to Settings so a pre-migration boot still syncs. CalDAV creds (collectionUrl + username + password) are ESP-only — they were never in Settings
- **Lasso supports mixed selection (this branch):** the lasso tool selects strokes AND text boxes together — `selectedStrokeIds` + `selectedTextBoxIds` are both populated on pen-up via `LassoSelectionLogic.selectedIds` / `selectedTextBoxIds` (centroid-in-polygon, band-agnostic). The `onSelectionChanged` callback carries a `ClipboardPayload(strokes, textBoxes)`. Drag-to-move translates both lists live (lasso-overlay paints transforming boxes at `(dragDx, dragDy)`; `transformingBoxIds` excludes them from `composeStaticBitmap` so they don't double-paint); pen-up commits via parallel `store.replaceStrokes` + `store.replaceTextBoxes` (single-threaded executor serializes them; NOT atomic across tables — matches the existing failure model). Cut / Copy / Delete fan out to both tables: `copySelection` writes the full payload; `deleteSelection` calls `store.deleteStrokes` and `store.replaceTextBoxes(boxIds, emptyList())`. Paste anchors the combined `(strokes + boxes)` payload on the tap via `combinedBounds.center`, preserving relative offsets, with fresh ULIDs per element, persisted via the Phase-4 batch path. Recognize / To-do still operate on `payload.strokes` only (boxes are bystanders, AC5.2); `SelectionMenuView.show` hides those two buttons on a boxes-only selection (AC5.1)
- **Lasso pill routing (Recognize vs. To-do, split by intent):** the two existing lasso-selection buttons now do their own things. **Recognize** = "convert handwriting to a text box on the page" — always runs ML Kit → `MainActivity.insertRecognizedAsTextBox` → `DrawView.prepareRecognizedTextBox` → `TextBoxEditOverlay`, restoring the 219f2dd behaviour for everyone, CalDAV-configured or not. **To-do** = "send handwriting as a CalDAV VTODO" — checks `secureCreds.caldavCreds()`; when configured, runs ML Kit → straight into `CalDavTaskSheet` (full-screen modal, peer of `SettingsView`; no intermediate pill — the sheet's editable SUMMARY is the confirmation surface) → on Send: `NotebookStore.enqueueCalDavTask` (writes a row to `caldav_outbox`) → `caldavDrainer.tryImmediately(uid, OPTIMISTIC_SEND_TIMEOUT_MS=500)` races the periodic drainer for an optimistic toast: `TryOutcome.Sent` → "Task created", `TryOutcome.Queued` → "Task queued". When To-do is tapped without CalDAV configured, the long-standing `SelectionActionLogic.todo` placeholder dialog tells the user where to go. Both lasso buttons call `selectionMenu.dismiss()` immediately on tap so the "N selected …" pill never lingers under the overlays. Routing decision lives at the call site (`onRecognize` / `onTodo`) via an `onText: (text, screenBounds) -> Unit` callback threaded through `showRecognizeFlow`/`runRecognize`. Hand-built iCalendar VTODO (no dav4jvm/ical4j) per defensive-coding ethos; V1 is single-collection PUT-only, no PROPFIND discovery / ETag tracking / scheduling. `onBackPressed` adds a branch for `caldavTaskSheet` ahead of the existing overlays
- **CalDAV VTODO provenance + recognized-text ATTACH + deep link (Feature 2):** lasso → To-do now stamps the source page onto the emitted VTODO. `MainActivity.openCalDavTaskSheet` reads the active notebook/page (cached in `refreshPageIndicator` as `activeNotebookId`/`activeNotebookName` — the single navigation chokepoint, no extra send-time store hop) and one off-thread `loadSettings` for the sync base, then hands a `CalDavTaskSheet.TaskContext(recognizedText, provenance, webUrl)` to the sheet. Provenance is a `VTodoProvenance` with `source = "lasso"` (`TASK_SOURCE_LASSO`, frozen UB contract, filterable as `?source=lasso`), the four structured `X-FORESTNOTE-*` fields, and `nativeUrl = ForestNoteLink.native(nb, pg)`; `webUrl = ForestNoteLink.web(syncServerUrl, nb, pg)` becomes the standard iCal `URL`. **Recognized handwriting rides as an inline `text/plain` ATTACH, NOT COMMENT** — third-party task apps (Tasks.org) surface attachments but ignore COMMENT, and DESCRIPTION belongs to the user's own typed note; ATTACH is opt-in via the sheet's "Attach full recognized text" checkbox (pure gate `CalDavTaskSheetLogic.recognizedTextToAttach`). UltraBridge de-bloats the inline base64 to a content store on ingest (keys on `ENCODING=BASE64`), serves it at a signed URL, and surfaces FILENAME/FMTTYPE in MCP `task.Attachments` — verified end-to-end against UB's real parser/de-bloat. **`forestnote://` deep links:** the manifest declares a `VIEW`/`BROWSABLE` intent-filter for scheme `forestnote` and `launchMode=singleTask` (a second Activity would open a second `NotebookStore` and break the single-writer invariant). `onCreate` (cold launch) and `onNewIntent` (warm) parse via `ForestNoteLink.parse` → `routeToDeepLink` (dismiss any overlay → `goToNotebookPage`); a cold-launch deep link overrides the start-view/anti-flash-Library decision. To-do `onSend` also resets the tool to Pen (Lasso → Pen clears the selection in one chokepoint) so the next stroke draws instead of re-lassoing. Settings → "Calendar (CalDAV tasks)" carries a note that the back-links only resolve against an UltraBridge server (other CalDAV servers still store + sync the task)
- **Anti-flash launch + overlay-paint discipline (extends [[viwoods-writing-overlay]]):** the Viwoods writing overlay composites ABOVE the View pipeline, so ANY `DrawView` push (`clearAll`, `fullRefresh`, `gcRefresh` — all three funnel through `backend.pushBackgroundBitmap` + `resetOverlay`) paints the editor bitmap on top of whatever is showing. Pattern is now applied in three places:
  - **Cold-launch into Library:** `Settings.startView` lives in the DB JSON blob (two off-thread hops — too slow to consult before `setContentView`), so we mirror it to a plain `forestnote_launch` SharedPreferences (`LAUNCH_PREFS`, key `KEY_START_VIEW = "start_view"`) on every toggle. `MainActivity.onCreate` reads the cache synchronously; if `LIBRARY`, it hides the navbar + DrawView via `View.INVISIBLE` and calls `openLibrary()` BEFORE the first paint frame. The authoritative `store.loadSettings` callback writes the cache through (idempotent); if it disagrees (user toggled since last launch), it reverses course with a one-launch flicker that self-corrects. Constants are duplicated in `MainActivity` and `SettingsView` companions on purpose — this is a one-key cache with no shared code. NEW persistent storage alongside the existing `EncryptedSharedPreferences` for credentials.
  - **Delete-while-Library-showing:** `confirmDeleteNotebook` / `confirmBulkDelete` / `confirmDeleteFolder` all route through `deferEditorReloadIfOverlayShowing()`: when the Library is up, it marks `editorLoaded = false` and the handlers skip `reloadCurrentPage()` (would paint the fallback notebook on top of the Library). `closeLibrary()` then does `drawView.clearAll()` in its `!editorLoaded` branch before `loadEditor()` to wipe any stale bitmap. Without this, deleting the active notebook briefly flashed the fallback's strokes on top of the Library and left a hard ghost.
  - **FontCatalog post-scan refresh:** the `/system/fonts` scan kicked off in `onCreate` calls `drawView.fullRefresh()` from `runOnUiThread` a few hundred ms later. That call is now gated on `editorLoaded && !libraryView.isShowing && !settingsView.isShowing && !recycleBinView.isShowing` — without the gate, the font-scan completion painted the editor's bitmap on top of the Library on launch-into-Library. Lossless skip: the next `composeStaticBitmap` on real editor reveal picks up the live `fontResolver` naturally.
  - **`revealEditorChrome()` helper** undoes the launch-time `INVISIBLE` cover; called from `closeLibrary` / `goToNotebook` / `goToNotebookPage`. Idempotent.
- **Library header is two rows (this branch):** Row A is a 44dp toolbar (`text_library_title` "Library" 20sp bold, clickable → goes to root from a folder; weight=1 spacer; the 8 toolbar cells: Settings / Sync / Search / divider / Select / Recycle / Folder ▾ / Notebook ▾). Row B is a 40dp `library_breadcrumb_row` (the `library_counts` cluster leftmost — reads as the row's library-state header; a 1dp vertical divider mirroring the Search/Select rule above; `btn_library_back` Up button gone-at-root; `breadcrumb_container`; trailing weight=1 spacer). The breadcrumb has its own row so a deep path can grow without squashing the toolbar. `BreadcrumbView` renders an italic dimmed `↳ Library root` placeholder at root (the visible "this is the breadcrumb area" indicator); inside a folder it DROPS the leading "Library" segment returned by `BreadcrumbLogic.segments` (the toolbar title + Up button cover "jump to root" and "one level up"). Segments are 18sp via `BreadcrumbView.BREADCRUMB_TEXT_SP` — `BreadcrumbLogic.kt` is unchanged, only the rendering policy moved.
- **`openEditOverlay` `onCommitted` hook (this branch):** `openEditOverlay(box, isNewBox, focusForEditing, onCommitted: (() -> Unit)? = null)` fires its hook AFTER `commitOverlayBox`. The Recognize path (`insertRecognizedAsTextBox`) uses it to call `toolBar.selectTool(Tool.Text)` so the user can immediately move/resize/edit the freshly placed text box — previously the tool stayed on Lasso and the next tap re-lassoed the new box. Cancelling the overlay leaves the tool on Lasso (the user is discarding, not interacting with a new box).
- **Release 1.0 / Settings → About (this branch):** `versionName = "1.0"`, `versionCode = 2` in `build-logic/forestnote.android.application.gradle.kts`. `SettingsView.wireAboutVersion` reads `packageManager.getPackageInfo(...).versionName` at bind time — never a hard-coded constant — so the row tracks the manifest. Defensive: any PackageManager failure degrades to plain "ForestNote" instead of crashing.
- **Sync-on-close dirty gate (this branch):** the in-Activity full-screen overlays (Library / Recycle Bin / Settings) never fire `onPause`, so without an explicit trigger the user's edits sit unsynced until the 30-min `SyncController` timer fires. `MainActivity.syncIfDirty()` is wired into `closeLibrary` and `closeRecycleBin`: it reads `Settings.syncOnClose` off-thread (default true) and, if enabled, calls `NotebookStore.countPendingOps` (now the RhizomeSync adapter's pending-ops count = the `rhizome_outbox` size, since the adapter prunes acked ops on `markAckedThrough`; returns 0 when sync is off — see core:format) and fires `syncController.syncNow()` ONLY when there are unacked ops. Both store hops are off-thread + main-thread-posted, so `syncIfDirty()` can't race the surrounding `gcRefresh`/UI work. `closeSettings()` deliberately does NOT call this — that path already calls `syncController.resume()`, which runs a fresh session immediately AND restarts the timer (a double round-trip would be wasteful, and the user is in sync-config land where their next action drives the ceremony). User-facing toggle is the `check_sync_on_close` row in Settings → Sync; off means falling back to the periodic timer + `onPause` only.
- **CalDAV offline outbox + drainer:** failed PUTs never lose tasks. UI enqueues into `caldav_outbox` (LOCAL-ONLY SQLite table, see core:format Gotchas); the body is FROZEN at enqueue while credentials/URL are resolved FRESH on every drain attempt from `SecureCredentialsStore` — that asymmetry is what lets a user fix their password and have stuck rows recover. `CalDavOutboxDrainer` mirrors `SyncController`'s shape on purpose (resume/pause/Mutex-serialised pass/30-min periodic timer, `StateFlow<DrainerStatus>`); anyone touching one should consider the other. Retry policy lives in pure `CalDavOutboxLogic`: 2xx → Success; **412 → Success** (idempotent create-retry pattern — RFC 7231 + Outlook CalDAV Synchronizer / Sabre DAV); 5xx / 429 / `TransportError` → `RetryLater` with exponential backoff 5s → 10s → 20s … capped at 1h (no jitter — single-user queue); other 4xx → `DeadLetter`. `tryImmediately` is the Send-side optimistic toast race against the drainer. `NetworkAvailabilityMonitor` (defensive `ConnectivityManager.registerDefaultNetworkCallback` wrapper, lifecycle-bound to `onStart`/`onStop`) kicks `drainNow()` when WiFi comes back — addresses the Delta-WiFi failure mode. **No WorkManager** — the AiPaper is foreground-driven; SQLite + lifecycle hooks + `NetworkCallback` cover the actual failure mode. **No "X tasks failed" snackbar** — failures surface only in Settings → "Queued tasks" (by deliberate scope choice during brainstorming)

## Invariants
- All coordinates in DrawView: virtual for storage, screen for bitmap rendering
- WritingBufferQueue released in onPause, reacquired in onResume (shared device resource)
- clearAll() only clears in-memory state; database clearing is caller's responsibility (via `store.clear`)
- Tool selection is pure logic in ToolSelectionLogic (testable without Android)
- `DrawView.mergeStrokes` is a pure companion function: loaded ink first, then session ink, deduped by ULID — so the async load never clobbers strokes drawn during the load gap
- `MainActivity.onDestroy` calls `store.shutdown()`, which drains pending saves before closing the driver
- Page/notebook index math is pure in `PageNavigationLogic` (testable without Android); `MainActivity` drives all switching/CRUD through `NotebookStore` and never touches `NotebookRepository` directly
- `goToPage`/`goToNotebook`/`reloadCurrentPage` share one idiom: `clearAll()` → `store.switch*`/`load` → `mergeLoadedStrokes` + `fullRefresh` + refresh indicator/label

## Gotchas
- ViwoodsBackend cast in onResume: only safe because isEInk guards it
- Crash handler writes to /sdcard/Download/ first, falls back to app filesDir
- E-ink mode disables animations and ripple effects to prevent ghosting
- **Boox/Onyx: any editor-obscuring surface MUST drop firmware render before it draws, or it opens INVISIBLY and wedges input.** Live firmware render globally suppresses normal EPD posting, so an overlay/dialog shown over the live-firmware (Pen) canvas is added to the view tree but never painted — invisible, on top, eating every touch while only firmware drawing works ("frozen except drawing"; only system Back recovers). New editor-direct surfaces need wiring: content overlays call `backend.setInputSuspended(true)` before `show()` + resume on close (see `openLibrary`/`closeLibrary`); dialogs are covered automatically by `onWindowFocusChanged` (they steal window focus) — but anything reached from an already-suspended overlay must NOT re-toggle suspend (desyncs on return — see `anyEditorObscuringOverlayShowing`). After a dialog dismiss, `backend.cleanNextReconcile()` clears its text ghost. Learned the hard way 2026-06-02 (`60ee6ad`); measured via on-device DIAG trace (taps reached handlers, panel never repainted). See memory boox-toolbar-coexistence.
- **Never call `drawView.clearAll()` / `fullRefresh()` / `gcRefresh()` while a non-editor View is on screen** (any dialog or full-screen overlay). All three funnel through `backend.pushBackgroundBitmap` + `resetOverlay`, and the Viwoods writing overlay composites ABOVE the regular View pipeline on this panel — so every one of them puts the editor bitmap on top of whatever else is showing. Safe contexts: editor↔editor transitions (page/notebook switch — funnel through `refreshPageIndicator`'s callback) and dialog/overlay DISMISS paths. `OcrTextDialog.show` documents this rule inline. The CalDAV-task-sheet and Recognize-pill paths inherit it; so do the new launch-into-Library cover (Key Decisions: Anti-flash launch), the delete-while-Library handlers (`deferEditorReloadIfOverlayShowing`), and the FontCatalog post-scan `fullRefresh` (gated on `editorLoaded && !libraryView.isShowing && !settingsView.isShowing && !recycleBinView.isShowing`). Learned the hard way building task #12, re-confirmed on the launch-flash and delete-ghost fixes
