# Format Domain (core:format)

Last verified: 2026-06-02 (phase8-rhizome-cutover)

## Purpose
Persists a notebook library to a single SQLite file (.forestnote) using SQLDelight. One library file holds `folder â†’ notebook â†’ page â†’ (stroke, text_box)` (folders are an optional hierarchy; notebooks/folders at the root have a NULL parent). Provides auto-save on pen-up, full restore on app launch, folder/notebook/page CRUD + active-context switching, Library card/thumbnail read paths, soft-delete (Recycle Bin) of notebooks/folders, and the global `Settings` blob + per-page template overrides â€” all behind `NotebookRepository`.

## Key Decisions
- SQLDelight over Room: type-safe SQL, smaller footprint, no annotation processing
- BLOB encoding for points: 5 ints per point (x, y, pressure, tsHigh, tsLow), little-endian. Compact and fast.
- Single library file `default.forestnote` holding `folder â†’ notebook â†’ page â†’ stroke`; an `app_state` singleton row (id=0) records the active notebook+page, plus `settings_json` (the `Settings` blob) and `clipboard_json` (persisted clipboard; column added in v5, wiring deferred)
- Settings stored as ONE JSON column (`app_state.settings_json`), not a key/value table: read on launch, written rarely, never queried by field. `Settings.json` codec uses `ignoreUnknownKeys` + `encodeDefaults` so the blob round-trips across build versions and adding a field never needs a migration
- Per-page template lives on `page.template`/`page.template_pitch_mm`. **Freeze-at-creation (B4):** pages always store a CONCRETE template â€” the global default seeds only at creation (new notebook's first page from `Settings`; later pages copy their predecessor's columns). Changing the global default never retroactively changes existing pages. Legacy `template IS NULL` pages are baked to the current default once on open (`bakeNullPageTemplates`, idempotent). NULL is a transient/legacy state only; the resolver (`page.template ?: default`) remains as a safety fallback
- `PRAGMA foreign_keys` is OFF, so `ON DELETE CASCADE` is inert â€” multi-row HARD deletes (page delete; future E4 permanent-delete) run through `db.transaction { }` children-first
- Soft-delete (Recycle Bin, AC7): notebook/folder deletes stamp `deleted_at`/`deleted_batch_id`/`deleted_root_id` rather than removing rows. Every query that must ignore tombstones reads the `notebook_live`/`folder_live` VIEWs (not the base table) so the filter can't be forgotten. Standalone notebook delete â†’ NULL batch/root; folder delete â†’ fresh ULID batch shared across the subtree, `deleted_root_id` = the tapped folder. Subtree is computed from LIVE folders BEFORE stamping (they're still live then). The folder cascade only re-tombstones LIVE notebooks (`softDeleteNotebooksInFolders â€¦ AND deleted_at IS NULL`) so an already-standalone-deleted notebook is NOT absorbed into the folder's batch
- Recycle Bin reads/ops live on the same `db.transaction` discipline: `recycleBinEntries` feeds tombstoned rows to `RecycleBinLogic`; `restoreEntry` clears tombstone columns (folder batch: folders first so notebooks reconcile against now-live folders via `RecycleBinLogic.restoreFolderFor`); `permanentlyDelete`/`emptyRecycleBin`/`purgeExpiredBinEntries` hard-delete (strokesâ†’pagesâ†’notebook, then folders). `Settings.recycleBinRetentionDays` (0 = never) drives the launch-time purge from `bootstrap()`
- Factory methods (open/forTesting/openExisting) instead of public constructor: controls lifecycle

## Invariants
- After open there is always â‰Ą1 notebook, each with â‰Ą1 page, and a populated `app_state`
- StrokeSerializer rejects truncated BLOBs (returns empty list, never crashes)
- Notebook/page/stroke ids are client-minted ULIDs (TEXT); strokes carry their id from creation, so saveStroke() does not return or change it
- Strokes load ordered by the explicit `z` column (ascending = paint order); saveStroke assigns z = max(z)+1 per page
- Stroke ops are scoped to `currentPageId`; switching notebook/page updates the current ids and persists them to `app_state`

## Gotchas
- NotebookRepository.open() silently deletes and recreates on any database error
- Ids are client-minted ULIDs minted at construction â€” there is no "unsaved" id state
- Schema is at version 18 (auto-derived from the seventeen `.sqm` files); `migrations/2.sqm` is a DESTRUCTIVE v2â†’v3 reset (drops existing data), consistent with `1.sqm`; `3.sqm`â€“`17.sqm` are non-destructive. `15.sqm`/`16.sqm` are local-only (last_page_id etc.); **`17.sqm` adds the SYNCED `notebook.aspect_long_axis`** (per-notebook page aspect, NULL = legacy 3:4) â€” the v3â†’v4 wire-hash bump, coordinated with UltraBridge
- Text-box sync is LIVE: `NotebookRepository.TEXT_BOX_SYNC_ENABLED = true` (Phase 6 cutover, wire schema v2) so `text_box` mutations call `enqueueOp` and `backfillOutbox` uploads pre-cutover boxes. Post-RhizomeSync-cutover the plumbing is structural, not hand-written: `text_box` is a table in `ForestNoteRegistry`, so `enqueueOp`â†’`syncStore.capture("text_box", id)` reads the committed row and the library's `WireCodec` encodes it; the apply side is the generic adapter upsert. (The old `SyncWire.textBoxCols`/`SyncMerge.knownCols["text_box"]`/`buildCols` machinery is gone.)
- Recognized-text round-trip (wire schema v3, hash `724411ebâ€¦`): `page_text_from_server` is a SERVER-AUTHORED-ONLY synced entity (pk = page ULID) â€” the UltraBridge server OCRs a synced page and pushes the text back; the client only ever APPLIES it, never authors it. Post-cutover this single-writer guarantee is STRUCTURAL in the registry: `ForestNoteRegistry` marks `page_text_from_server` `serverAuthoredOnly`, so the adapter's `capture` is a no-op for it and the client can't put it on the wire. The client DOES locally mark its mirror copy stale via the LOCAL-ONLY `stale_at` column (added in `12.sqm`, NOT a registry column, so the wire hash is unchanged): every ink/text-box mutation in `NotebookRepository` (saveStroke, deleteStroke, applyErase, clearPage, saveTextBox, deleteTextBox) calls the private `markPageOcrStale(pageId, now)` helper INSIDE its own transaction, and the `applySyncOps` post-apply hook clears `stale_at` via `clearPageTextStale` whenever a `page_text_from_server` op is applied (the generic adapter upsert can't touch a non-registry column). `page_text_from_client` is the RESERVED client-authored sibling â€” its columns are in the registry (so in the v3 hash; no future bump) but nothing authors it yet. `SYNC_BACKFILL_VERSION` stays 1. On-device search reads `page_text_from_server.text` via `searchPageOcrText` (LIKE-based â€” no FTS index yet; revisit if latency on a representative corpus exceeds ~500 ms)
- LOCAL-ONLY tables/columns (NEVER on the UB sync wire â€” same structural discipline): `page_text_from_server.stale_at` (`12.sqm`), the entire `caldav_outbox` table + `caldav_outbox_drain` index (`13.sqm`), AND `sync_state.rhizome_migrated`/`stored_schema_hash` (`14.sqm`). None of these are declared in `ForestNoteRegistry`, so the v3 wire hash (`724411ebâ€¦`) is unchanged across every bump. Rule of thumb when adding LOCAL-ONLY storage: bump the schema version (so existing devices migrate forward), keep the body frozen in the local row, do NOT add it to `ForestNoteRegistry` (the registry IS the wire schema â€” anything in it changes the hash), and call it out here so future sync work doesn't accidentally adopt it
- Sync capture (RhizomeSync, post-Phase-8): every mutating repo method calls `enqueueOp(table, pk, now)` INSIDE its own transaction; `enqueueOp` is now a thin shim = `runBlocking { syncStore.capture(table, pk) }` (the ~20 call sites are unchanged). The adapter re-reads the committed row, encodes it via the registry + `WireCodec`, appends to `rhizome_outbox` and stamps `rhizome_row_meta` + an HLC `op_ts`. When sync is OFF (no `site_id` minted in `rhizome_sync_state`) `capture` is a no-op, so sync ships dormant. `enableSync()`â†’`mintSiteId()` + `backfill()` (re-captures EVERY row â€” idempotent under server LWW; re-uploads pulled rows once per join/schema-bump, an accepted behavior change from the old `lacksMeta`-skipping backfill). NOT captured: `modified_at` touches (not a registry column), permanent-delete/purge paths (local reclaim), bootstrap/recovery inserts (deferred to the join handshake so a fresh device pulls before uploading its auto-created notebook), and `page_text_from_server` (registry `serverAuthoredOnly`)
- VIEWs (`notebook_live`/`folder_live`) are NOT auto-added to migrations by SQLDelight â€” only to `Schema.create()`. Any migration that introduces or must preserve a view has to `CREATE VIEW` it explicitly, or upgrading devices hit "no such table: â€¦_live" at runtime (regression-tested in `MigrationTest.v6ToV7â€¦`)
- `folder`/`notebook` row lookups use SQLite `IS ?` (not `= ?`) when scoping by parent/folder id so a NULL bind matches root-level rows; `= ?` would never match NULL
- `upsertAppState` is an `ON CONFLICT(id) DO UPDATE` upsert, NOT `INSERT OR REPLACE` â€” REPLACE would delete+reinsert the row and reset `settings_json`/`clipboard_json` to their defaults on every page switch (regression-tested in SettingsStorageTest)
- `notebook.modified_at` is bumped (via `touchCurrentNotebook`) inside every ink-mutating transaction â€” saveStroke, deleteStroke, applyErase (so app-layer cut/delete/paste/move inherit it), clearPage â€” using an injectable clock
- Do not rely on FK cascade for deletes â€” `PRAGMA foreign_keys` is off; use transactional children-first deletes
